3. How Proofloop becomes an adaptive Fusion model router

Yes — your Proofloop can become a production adaptive router. Do not start with full RL. Start with a contextual router + proof feedback loop.

The router’s job:

For this task, choose the cheapest model/tool plan that is likely to pass the verifier, and escalate only when proof says it is necessary.

You already have the right direction: the BTB live-browser matrix ultimately produced 100/100 fresh-room chat receipts, with the important caveat that completion/product-path proof must be kept separate from official semantic score claims.

Fusion router loop
1. Classify task.
2. Choose cheapest credible route.
3. Run NodeAgent.
4. Verify with Proofloop:
   - deterministic scorer
   - evidence verifier
   - export/reopen
   - visual judge
   - cost/latency
5. If pass:
   - store route as successful sample
6. If fail:
   - classify failure
   - escalate model/tool plan
   - rerun
7. Periodically update router policy from stored traces.

This aligns with current research on agent trace assurance: modern agent failures include unsupported claims, non-termination, external side effects, role drift, and tool/memory faults, so trace contracts and machine-checkable step verdicts are necessary for evaluating and improving orchestration.

online RL directly changing production behavior

Use:

  router recommends route but current route runs

  route 10-20% of low-risk tasks through Fusion

  router can escalate on verifier failure

prod mode:
  router is default, but hard-capped by credits and safety gates

CATEGORY 1 — QUALITY IMPROVEMENT LOOPS (Make the output better before it leaves the system)

1. Generate → Critique → Rewrite

The most important loop in AI engineering.
Generate output. Critic reviews it. Generator rewrites based on feedback. Repeat until quality threshold is met.
Not one model. Two roles. One pipeline.
plaintext
[Generator] → draft
[Critic]    → "paragraph 3 is vague. missing evidence. tone is off."
[Generator] → rewrite based on critique
[Critic]    → "better. but conclusion still weak."
[Generator] → final rewrite
Used for: writing, code review, reports, strategy docs, sales emails.
The insight: the model that generates is not the best judge of its own output.
A separate critic finds what the generator missed every time.
Image
2. Score-and-Retry Loop

Generate. Score. Retry if below threshold.
Simple. Powerful. Underused.
score = evaluate(output)
plaintext
score = evaluate(output)

while score < threshold:
    output = generate(prompt)
    score = evaluate(output)
    attempts += 1
    if attempts > max_retries:
        return best_so_far
Best when quality is measurable — extraction accuracy, format compliance, factual correctness, lead scoring.
The generator doesn't know it's being graded.
The evaluator does.
That separation is the pattern.
3. Multi-Critic Loop

One critic has blind spots.
Use four.
→ Correctness critic: is it factually accurate? 
→ Style critic: is it clear and well-written? 
→ Safety critic: is it appropriate and safe? 
→ Domain critic: does it meet specialist standards?
Each evaluates independently.
Final output must satisfy all four before it exits.
Used in: medical AI, legal document review, financial analysis, regulated content.
Image
4. Adversarial Critique Loop

The critic's only job is to break the answer.
Not improve it. Break it.
Questions the adversarial critic asks:
→ What assumptions fail here? → What evidence is missing? → What would a skeptic say? → Where is this confidently wrong?
The generator then defends or rewrites.
The best answer survives the attack.
Used for: research synthesis, investment thesis review, strategic planning, risk analysis.
5. Judge Ensemble Loop

One judge gives noisy scores.
Five judges average out the noise.
Run the same output through multiple evaluators.
Aggregate scores.
Only outputs with high consensus advance.
Used when: single-model evaluation is unreliable, stakes are high, edge cases matter.
Image
CATEGORY 2 — MEMORY LOOPS (Learn from what happened so next time is smarter)

6. Reflexion Loop

The most important self-improvement pattern that exists.
Agent fails. Agent analyzes why it failed. Agent stores the lesson. Agent retries with that lesson in context.
Each iteration: smarter than the last.
plaintext
attempt 1: fails
reflection: "I assumed X but X was wrong. Next time verify X first."
attempt 2: incorporates lesson → partial success
reflection: "Better. But I skipped Y. Add Y check."
attempt 3: succeeds
The difference between a system that fails once and a system that only fails once.

Image
7. Memory Update Loop

After every task, store three things:
→ What decision was made → What the outcome was → What would be done differently
Future runs inherit this knowledge.
The system in month 6 is not the same as the system in month 1.
It has read 6 months of its own history.
8. Error Library Loop

Store every failure.
Wrong answer. Bad output. Failed execution. Edge case.
Before acting on a new task:
Search the error library first.
If a similar failure exists → apply the known fix before even starting.
The system stops making the same mistake twice.
The most underused pattern in production AI.
9. Success Pattern Loop

Most engineers only store failures.
Store successes too.
When a task goes well:
→ Save the approach → Save the context → Save what made it work
Retrieve successful patterns when facing similar tasks.
Learn from wins. Not just mistakes.
10. Memory Compression Loop

Memory grows forever.
Unlimited memory is unusable memory.
After N items accumulate:
Compress them.
Many specific memories → fewer higher-level abstractions.
plaintext
Before compression:
"Failed on task A because of X"
"Failed on task B because of X"  
"Failed on task C because of X"

After compression:
"Pattern: X causes failures. Always check X first."
Context stays manageable. Patterns stay accessible. System stays fast.
Image
CATEGORY 3 — PLANNING LOOPS (Adapt the plan when reality changes)

11. Plan → Execute → Replan

The most common mistake in AI agent design:
Treating the plan as fixed.
Plans break on contact with reality.
The pattern:
Create plan → execute step → observe outcome → update plan → continue
Not a waterfall.
A spiral.
Each lap around tightens the approach.
Used when: environment changes, tasks have dependencies, long-horizon goals.
Image
12. Dynamic Workflow Loop

Most pipelines are fixed.
Step 1 → Step 2 → Step 3. Always.
Dynamic workflows change based on results.
If output A → run branch X If output B → run branch Y If output C → skip to step 5
The pipeline decides its own shape at runtime.
Used in: multi-document research, customer support routing, adaptive content pipelines.
13. Goal Decomposition Loop

Large goal enters.
System breaks it into subgoals.
Each subgoal breaks into tasks.
Each task breaks into steps.
Decompose until each unit is small enough to execute in one call.
plaintext
Goal: "Write a comprehensive competitive analysis"
↓
Subgoal 1: "Identify top 5 competitors"
Subgoal 2: "Analyze each competitor's product"
Subgoal 3: "Compare pricing models"
Subgoal 4: "Identify gaps"
↓
Each subgoal → tasks → individual model calls
The loop keeps decomposing until the system can act.
14. Progress Evaluation Loop

Every N steps: stop and ask.
"Are we actually getting closer to the goal?"
If yes: continue current strategy. If no: change strategy, tools, or plan.
The system monitors its own progress.
Not just executes blindly.
Used in: long-running research agents, multi-day autonomous tasks, debugging agents.
15. Constraint Satisfaction Loop

Keep running until all constraints are met.
plaintext
while not all_constraints_satisfied(output):
    output = improve(output, unsatisfied_constraints)

constraints = [
    budget_under_limit,
    quality_above_threshold,
    latency_under_200ms,
    tone_matches_brand,
    no_hallucinations
]
Very common in production systems.
The output is not done until every business rule passes.
Image
CATEGORY 4 — EXPLORATION LOOPS (Find the best answer by trying multiple paths)

16. Branch-and-Explore Loop

Don't commit to one path.
Explore several simultaneously.
plaintext
paths = [
    generate(approach="conservative"),
    generate(approach="aggressive"),
    generate(approach="creative")
]

scores = [evaluate(p) for p in paths]
best = paths[scores.index(max(scores))]
Compare outcomes. Choose the best branch. Discard the rest.
Used for: content variations, architecture decisions, debugging multiple hypotheses, A/B generation.


Image
17. Tree Search Loop

Branch-and-Explore goes one level deep.
Tree Search goes as deep as needed.
Expand the most promising nodes. Prune the weakest ones. Keep exploring until the solution is found.
plaintext
root → [A, B, C]
A → [A1, A2]   # A looks promising, expand it
B → prune      # B is weak, stop here
A1 → [A1a, A1b]
A1a → solution ✓
Used for: complex reasoning chains, multi-step planning, code debugging, research synthesis.
Computationally expensive but finds solutions single-pass calls cannot.
18. Debate Loop

Two agents. One topic. Opposite positions.
Agent A argues for the answer. Agent B argues against it.
Each round challenges assumptions, demands evidence, exposes weak logic.
Final answer emerges through disagreement.
Not through agreement.
The adversarial pressure finds what confident single-agent answers miss.
Used for: investment decisions, strategic planning, risk assessment, research critique.

CATEGORY 5 — SYSTEM OPTIMIZATION LOOPS (The loop improves the loop)

19. Prompt Optimization Loop

Most engineers write a prompt once and never touch it again.
Prompt optimization loops change that.
The system:
→ Runs the prompt on a test set 
→ Scores every output 
→ Identifies where the prompt fails 
→ Rewrites the prompt to fix those failures → Reruns and rescores
The prompt gets better automatically.
Without a human touching it.
plaintext
current_prompt = "Summarize this document."

for iteration in range(max_iterations):
    outputs = [run(current_prompt, doc) for doc in test_set]
    scores = [evaluate(o) for o in outputs]
    avg_score = mean(scores)
    
    if avg_score >= target:
        break
    
    failures = [o for o, s in zip(outputs, scores) if s < threshold]
    current_prompt = improve_prompt(current_prompt, failures)
    # Prompt rewrites itself based on where it fails
Used in: production pipelines, automated content systems, classification tasks.
The best prompts in production AI were not written by a human.
They were evolved.

20. Workflow Optimization Loop

This is where it gets interesting.
The loop improves the loop.
The system measures its own performance:
→ latency: how long does each step take? 
→ cost: how many tokens does each call use? 
→ quality: what is the output score at each stage?
Then it modifies its own workflow.
Too slow? Parallelize two steps. Too expensive? Replace a GPT-4 call with a smaller model where quality holds. Quality dropping? Add a critic before the final output.
plaintext
metrics = measure_workflow(outputs, latency, cost)

if metrics.latency > target_latency:
    workflow = parallelize(slow_steps)

if metrics.cost > budget:
    workflow = replace_with_cheaper_model(high_cost_steps)

if metrics.quality < threshold:
    workflow = add_critic_before(final_output_step)
This is where truly self-improving systems begin.
Not just outputs that improve.
Systems that redesign themselves.

2. Where the 20 loops fit

Not every loop should run on every task. The mistake would be to turn every user request into an expensive 20-loop monster. Instead, NodeRoom should pick loops by task class, risk, budget, and proof requirement.

Category 1 — Quality loops

These improve the answer before the user sees it.

1. Generate → Critique → Rewrite

Use for:

diligence memo
profile research notebook
accounting explanation
investor-ready summary
follow-up email draft

NodeRoom implementation:

NodeAgent drafts output
  -> EvidenceCritic checks source coverage
  -> FinanceCritic checks domain correctness
  -> UXCritic checks readability
  -> NodeAgent rewrites

Good for notebook / Tiptap / Notion-style outputs.

2. Score-and-Retry

Use for:

SpreadsheetBench
BankerToolBench
accounting reconciliation
cash-flow statement
AR/AP aging
export/reopen package

NodeRoom implementation:

candidate workbook
  -> deterministic scorer
  -> if score < threshold:
       retry with failure summary
  -> save best attempt

This is exactly what your fresh-room benchmark proof system is moving toward: no claim without a runtime receipt, screenshot/video, verifier, output paths, and pass criteria.

3. Multi-Critic

Use for high-stakes outputs:

finance
accounting
banking
VC diligence
legal/compliance-adjacent work

Critics:

Accounting critic:
  Are numbers and debits/credits correct?

Evidence critic:
  Does every claim cite source evidence?

Visual critic:
  Is the UI state legible?

Safety critic:
  Any private leak / overclaim / unsupported claim?

Cost critic:
  Did this use too expensive a route?
4. Adversarial Critique

Use for:

investment thesis
risk memo
credit memo
dealflow prioritization
portfolio company profile

Prompt role:

Break this answer.
What assumption fails?
What source is missing?
What would a skeptical banker / investor / auditor reject?

This matters because finance users do not trust polished prose. They trust objections survived.

5. Judge Ensemble

Use sparingly because it costs more.

Use when:

client-facing report
benchmark proof
public demo
high-stakes financial output
landing-page/product taste review

Example:

deterministic scorer
+ Gemini visual judge
+ evidence verifier
+ human reviewer

This is also how your Feature Walkthrough / visual judge layer becomes a reward signal instead of just a demo asset.

3. Memory loops

This is where NodeMem becomes essential.

6. Reflexion Loop

This is your default repair pattern.

attempt fails
  -> classify why
  -> summarize what failed
  -> store failure memory
  -> rerun with that lesson

Example:

Failure:
  source_open_literal missing sourceArtifactId

Memory:
  "Do not call source_open_literal until capture_source returns a sourceArtifactId."

Next attempt:
  use capture_source first

This is how the agent stops repeating tool misuse.

7. Memory Update Loop

After every meaningful run, store:

decision made
outcome
what to do differently
source evidence
model/tool route
cost

For NodeRoom:

NodeMemEpisode
NodeMemFact
NodeMemFailurePattern
NodeMemContextPack

NodeMem recently moved closer to real production usage because public room chat can now record memory episodes, while private/system messages are excluded and recording runs out-of-band so it cannot roll back the send path.

8. Error Library Loop

This should become one of NodeRL’s highest-value features.

Before starting a task:
  search similar failures

If match:
  inject known fix before the first attempt

Examples:

empty managed-write args
volatile receipt path
passive job starvation
no sourceArtifactId
Gemini key missing
chart visual grade missing

The passive intelligence workpool saturation incident is the canonical error-library item: passive jobs auto-fired, retried, saturated workpool slots, and starved foreground user jobs. That failure should permanently shape the loop policy.

9. Success Pattern Loop

Store wins too.

Example:

Task:
  BTB package generation

Successful path:
  upload official inputs
  public @nodeagent
  Focus Mode
  package verifier
  export/reopen
  durable evidence copy
  Gemini attach

Next similar task should reuse this path before exploring.

10. Memory Compression Loop

NodeMem cannot keep dumping everything into the prompt.

Compression ladder:

raw trace
  -> attempt summary
  -> failure pattern
  -> strategy rule
  -> routing policy

Example:

100 raw failures:
  "source open failed because sourceArtifactId missing"

compressed rule:
  "All source-open tools require capture_source provenance first."

This is how NodeMem stays useful instead of becoming prompt bloat.

14. Progress Evaluation Loop

This is what prevents 17-hour runs from becoming blind loops.

Every N steps:

Are we closer to the goal?
What output exists?
What evidence exists?
What is still missing?
Are we repeating a failed path?
Should we switch model/tool?
Should we stop and return partial?

This should be a NodeTrace/NodeLoop primitive.

15. Constraint Satisfaction Loop

This is the finance-grade loop.

The output is not done until constraints pass:

all required fields filled
every material claim cited
no unsupported final facts
spreadsheet formulas recompute
exports reopen
trace exists
visual state legible
cost under cap
no private leak
human edits preserved

This maps directly to your proof registry and fresh-room acceptance matrix. The matrix already formalizes proof signals like fresh-room join, public @nodeagent, streaming progress, trace/video artifacts, no memory-mode shortcut, focus boxes, evidence anchors, mutation visibility, export/reopen, and official scorer handoff.

6. System optimization loops

This is where NodeRL and Fusion router become real.

19. Prompt Optimization Loop

Use internally.

But do not let prompt optimization silently modify production prompts without proof.

20. Workflow Optimization Loop

This is the big one.

The loop improves the loop.

If latency too high:
  parallelize source capture

If cost too high:
  route extraction to cheaper model

If evidence score low:
  add source capture before synthesis

If visual quality low:
  add screenshot judge before final

If retry storms appear:
  lower attempts, quarantine provider, change tool schema

If passive suggestions are noisy:
  switch to digest mode / suppress signal

This is your adaptive Fusion router.

7. How this becomes NodeRL

NodeRL is the system that turns these loops into data.

Loop execution
  -> NodeTrace records trajectory
  -> NodeEval creates reward
  -> NodeMem stores memory
  -> NodeLoop decides repair
  -> Fusion router updates policy

In RL terms:

state:
  room, artifact, task, UI, memory, evidence, budget

action:
  model route, tool call, write, ask, retry, stop

observation:
  tool result, screenshot, artifact diff, user-visible state

reward:
  scorer, visual judge, evidence, cost, latency, human accept

next state:
  updated room, updated artifact, updated memory

That is why Joseph’s point about agentic RL is important: your loops produce the missing training substrate.

8. How Convex fits the loop

Convex gives you the durable loop rails:

Mutation:
  fast transactional write
  create job
  append trace event
  write evidence fact
  update artifact

Action:
  call LLM
  call Linkup
  call Firecrawl
  run external verifier
  run model-router experiment

Query:
  render live room state
  stream progress
  show trace/storybook
  show cost/reward state

Do not put long-running LLM or web work inside mutations. Keep the loop as:

mutation creates checkpoint
  -> action performs nondeterministic work
  -> internal mutation commits result
  -> query streams updated UI

This is the safe Action/Mutation/Query contract.

Also: use append-only trace rows rather than hot-patching one agentJobs row for every stream event. The prior OCC storm is exactly why loop traces need append-only event tables and separated workpools.

14. The biggest rule: loops must be bounded

The article is right that loops make agents improve. But your real-world lesson is:

Unbounded loops become cost explosions, retry storms, and invisible user frustration.

So every loop needs:

max attempts
max cost
max time
progress check
failure taxonomy
human escalation
receipt

The old passive room intelligence failure happened because a background loop lacked the right boundaries and priority isolation. It created passive jobs, retried on spend/rate limits, and starved explicit user work. That is exactly what loop engineering must prevent.
